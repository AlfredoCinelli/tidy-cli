Explanation

Understanding the concepts and philosophy behind Tidy CLI.

Why Tidy CLI Exists

The Problem

Modern Python development involves multiple quality assurance tools:

  • Ruff for linting and formatting
  • MyPy for static type checking
  • Pydoclint for docstring validation
  • Pytest for testing with coverage

Each tool requires separate installation, configuration, and execution. Developers often struggle with:

  • Remembering different command syntaxes
  • Managing tool-specific configurations
  • Ensuring consistent execution across team members
  • Setting up CI/CD pipelines with multiple tool calls

The Solution

Tidy CLI unifies these essential tools under a single, intuitive interface. Instead of running:

ruff check src/ --fix
ruff format src/
mypy src/
pydoclint src/
pytest tests/ --cov=src --cov-report=html

You simply run:

tidy-cli lint run --fix
tidy-cli pytest run

Design Philosophy

Simplicity First

Tidy CLI prioritizes ease of use over extensive customization. The tool makes opinionated choices about:

  • Default configurations: Sensible defaults that work for most Python projects
  • Command structure: Intuitive verb-noun patterns (lint run, pytest run)
  • Output formatting: Clean, readable results from all tools

Configuration Over Convention

While providing defaults, Tidy CLI respects your existing configurations:

  • Uses your pyproject.toml settings for each tool
  • Allows path customization for different project structures
  • Supports skipping tools that don't fit your workflow

Workflow Integration

Tidy CLI is designed to fit seamlessly into existing development workflows:

  • Local development: Quick quality checks during coding
  • Pre-commit hooks: Automated fixing before commits
  • CI/CD pipelines: Consistent quality gates
  • IDE integration: Works with existing tool configurations

Tool Selection Rationale

Why Ruff?

Ruff was chosen as the primary linter and formatter because:

  • Speed: 10-100x faster than traditional Python linters
  • Comprehensive: Combines functionality of multiple tools (flake8, isort, black)
  • Modern: Built with Rust for performance and reliability
  • Configurable: Extensive rule selection and customization

Why MyPy?

MyPy remains the gold standard for Python static type checking:

  • Mature ecosystem: Extensive type stub library support
  • Gradual typing: Works with existing codebases
  • IDE integration: Excellent editor support
  • Community adoption: Widely used and trusted

Why Pydoclint?

Pydoclint ensures documentation quality:

  • Docstring validation: Checks parameter and return documentation
  • Style consistency: Supports multiple docstring conventions
  • Error detection: Finds mismatched signatures and documentation
  • Lightweight: Focused tool without feature bloat

Why Pytest?

Pytest is the de facto standard for Python testing:

  • Powerful fixtures: Flexible test setup and teardown
  • Plugin ecosystem: Extensive third-party extensions
  • Coverage integration: Built-in support for coverage reporting
  • Readable output: Clear test results and failure reporting

Architecture Overview

Project Structure

Tidy CLI expects a conventional Python project structure:

project/
├── src/           # Source code
├── tests/         # Test files  
├── local/         # Tidy CLI settings
└── pyproject.toml # Tool configurations

This structure promotes:

  • Separation of concerns: Clear boundaries between source and tests
  • Tool compatibility: Standard layout works with all integrated tools
  • Scalability: Structure scales from small scripts to large applications

:material-settings: Configuration Management

Tidy CLI uses a two-tier configuration system:

  1. Tidy CLI settings (local/tidy_cli_settings.json): Path and execution preferences
  2. Tool configurations (pyproject.toml): Individual tool settings

This separation allows:

  • Project-specific paths: Adapt to different project layouts
  • Tool independence: Maintain existing tool configurations
  • Team consistency: Share Tidy CLI settings while preserving individual tool preferences

Execution Model

Tidy CLI follows a simple execution model:

  1. Load settings: Read project-specific configuration
  2. Resolve paths: Determine target files/directories
  3. Execute tools: Run tools in logical order
  4. Aggregate results: Combine outputs into unified report

Comparison with Alternatives

vs. Make/Scripts

Traditional approach: 

make lint
make test
make format

Tidy CLI advantages: - No Makefile maintenance - Cross-platform compatibility - Consistent interface - Built-in tool integration

vs. Pre-commit

Pre-commit is complementary, not competitive:

  • Pre-commit: Git hook automation
  • Tidy CLI: Development workflow integration

Use both together: 

# .pre-commit-config.yaml
- repo: local
  hooks:
    - id: tidy-cli
      entry: tidy-cli lint run --fix

vs. All-in-One Tools

Some tools attempt to replace multiple quality tools:

Tidy CLI philosophy: - Best-of-breed: Use the best tool for each job - Flexibility: Swap tools as ecosystem evolves - Transparency: Clear understanding of what runs

Performance Considerations

Execution Speed

Tidy CLI optimizes for developer productivity:

  • Parallel execution: Tools run concurrently when possible
  • Smart caching: Leverages individual tool caching
  • Incremental runs: Only processes changed files when supported

Resource Usage

Resource consumption is managed through:

  • Tool selection: Choosing performant underlying tools
  • Process management: Efficient subprocess handling
  • Memory awareness: Avoiding unnecessary data duplication

Future Directions

Extensibility

Future versions may support:

  • Plugin system: Custom tool integration
  • Configuration profiles: Environment-specific settings
  • Advanced workflows: Complex execution patterns

Community

Tidy CLI aims to:

  • Gather feedback: Learn from real-world usage
  • Evolve standards: Adapt to Python ecosystem changes
  • Maintain simplicity: Resist feature creep while adding value

The tool's success depends on balancing power with simplicity, ensuring it remains approachable for new developers while providing value for experienced teams.
